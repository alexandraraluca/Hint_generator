Codeforces Round 907 (Div. 2) Editorial
By 127.0.0.1, history, 2 years ago, translation,

Thank you for participating!

1891A - Sorting with Twos

Look at the difference array bb. bibi = ai+1−aiai+1−ai for each i<ni<n.

If array is sorted, its difference array has all of its elements non-negative.

And operation is adding 11 to bibi if ii is a power of 22.

So look at difference array. If there is such ii that ii is not a power of 22 and bi<0bi<0 then answer is «NO». Else answer is «YES».

Time complexity: O(n)O(n).

1891B - Deja Vu

Let a number be divisible by 2x2x. Then after applying the operation it is no longer divisible by 2x2x. From this we can conclude that if we apply the operation xixi, and there is such an operation j<ij<i that xj<xixj<xi, then the operation xixi does not change the array. So, it is useless and can be simply not processed.

Then we will maintain the minimum of the processed xx. If the new operation is smaller than the minimum processed one, we will process the operation and update it. Otherwise we just won't do anything. Since the minimum processed will decrease at most 3030 times, the time complexity is O(30⋅n)O(30⋅n).

1891C - Smilo and Monsters

Note that if the second operation were free, we would need ⌈sum2⌉⌈sum2⌉ operations to get rid of all the monsters. Indeed, when we kill one monster, we can kill a second monster for free with a second operation. But the second operation is not free. So we need to use the second operation as little as possible.

To do this, we need to apply ultimates (second attack) on the current largest horde by number of monsters, when the combo counter reaches the size of the largest horde. And we apply the first attack on the smallest hordes. This is so because the combo counter allows us to defeat ⌈sum2⌉⌈sum2⌉ monsters. But since we can't apply this operation on several hordes at once, we need to keep the number of hordes on which we apply these attacks of the second type as small as possible. Then we can use the greedy algorithm described above. Formally, we need to keep a sorted array and store two pointers: pointer ii — to the smallest horde, jj — to the largest horde. Until ii is equal to jj: if after destroying all horde ii we can't kill horde jj with ultimates, we destroy horde ii, increase pointer ii by 11 and increase combo counter xx by aa[ii]. Otherwise, hit the horde ii so many times that the combo counter xx becomes aa[jj]. Then apply a second attack on horde jj, reduce horde jj's counter by 11, reduce aa[ii], and nullify the combo counter. When ii becomes equal to jj, you just need to apply the first attack the right number of times to finish with ultimates (or not if aa[ii] == 11).

Total complexity OO(nlognnlogn).

1891D - Suspicious logarithms

First, let us estimate the number of such ii that g(i)≠g(i+1)g(i)≠g(i+1). This can happen if f(i)≠f(i+1)f(i)≠f(i+1), but it happens rarely (about lognlogn times). And if f(i) is equal, we can see that f(i)≥2f(i)≥2 for any 4≤i4≤i, and on a segment with equal f(i)f(i) transitions will be O(logn)O(logn), where nn is the length of the segment. That is, there are O(logn)O(logn) segments with equal f(i)f(i), and on each segment there are at most O(logn)O(logn) transitions. Then there are a total of O(log2C)O(log2C) transitions, where CC is the maximum value of ii in the problem. In fact, in reality, there are even fewer transitions - something like 6060. Now we can suppose all such ii that g(i)≠g(i+1)g(i)≠g(i+1), and by passing over them we can consider the sum of g(i)g(i) for all i≤ri≤r. Or you can write a binary search and read the answer even faster.

1891E - Brukhovich and Exams

Consider that every ai≠1ai≠1. Then, we can split our array into blocks in which each adjacent element has gcd=1gcd=1. For example, consider a=a=[22, 33, 44, 55, 55, 66, 77], then the array divides into two blocks: [22, 33, 44, 55] and [55, 66, 77]. Notice that if the lenght of a block is lenlen, then it gives len−1len−1 sadness to Smilo. Also notice that we can reduce sadness by two using only one operation: we can, for instance, nullify the second element in a block (if the length of the block is more than two). For illustration, if we nullify the second element in block [55, 66, 77], the sadness will decrease by two, and the block will be splitted into [55], [00], [77] with zero sadness in each block. If the size of a block is exactly two, we can decrease the sadness by one using one operation. Then notice that while we are having at least one block, which lenght is at least 33, we can decrease sadness by 22. Otherwise, we can decrease sadness at most by 11.

Unfortunately, we have ones in our array. In this case our algorithm won't work, since in order to make pair [11, 11] good, we will have to nullify both elements. Then let's slightly modify the algorithm: we will put consecutive ones in different blocks. For example, if a=a=[11, 11, 11, 33, 44, 44, 11], then our array will be splitted into 4 blocks: [11, 11, 11], [33, 44], [44], [11]. Now notice that if the block of ones is located at the edge of our array, then the sadness will decrease by lenlen using lenlen operations (lenlen is the lenght of the block). Otherwise, sadness will decrease by len+1len+1 using lenlen operations. The final solution becomes obvious. Firstly, we should nullify the elements in non-ones blocks (if possible) in order to decrease sadness by 22 each operation. Then we need to nullify whole blocks of ones (which are not on the edge of the array) in ascending order of lenght. And finally, we need to use remaining operations to reduce sadness by one each operation.

The complexity is OO(nlognnlogn).

1891F - A Growing Tree

Let's parse all the queries and build the tree. We can easily support subtree addition queries using segment tree on Euler tour of the tree. And when we add new vertex, we just need to set its value to zero. How to do it? You can get the value in this vertex now (by get query from the segment tree), let it be xx. Then you add −x−x to its value.

You can also use Fenwick tree to handle range add point get queries.

Time complexity: O(qlogq)O(qlogq).

Bonus: try to find a solution using sqrt decomposition. I hope it will get TLE, but still.

      I believe that my solution 230576374 is the same as in this comment, so I'll describe it.
      Let's build our tree and for each vertex save its creation time and updates with current time and value.
      What is answer for some vertex? It's sum of all requests which were made on this vertex or any of parents LATER than creation time of current vertex. So let's create BIT for sum of requests by time and dfs our tree. We perform updates before getting ans and rollback them before dfs exit thus for each vertex only relevant updates remain:

      dfs(u, p):
    for (time, value) in updates[u]:
        bit.update(time, value)
    ans[u] = bit.get(createdAt[u], n)
    for (int v: g[u]): if v != p:
        dfs(v, u)
    for (time, value) in updates[u]:
        bit.update(time, -value)

      Each update is made exactly twice, so it's O((n+q)log(n))O((n+q)log⁡(n))

2nd ques. can be done in less than O(30.N) if we maintain another array for counting the what should be added to the array element if it is divisible by some power of 2.

class Codechef {
public static void main(String[] args) throws IOException {
        Scanner ab = new Scanner(System.in);
        int t = ab.nextInt();
        while(t-- > 0) {
           int N = ab.nextInt();
            int Q = ab.nextInt();
            long[] a = new long[N];
            for(int i=0; i<N; i++) a[i] = ab.nextInt();

           boolean[] flag = new boolean[31];
           int mini = 31;
           for(int i=0; i<Q; i++) {
               int pow = ab.nextInt();
               if(pow < mini) {
                   mini = pow;
                   flag[pow] = true;
               }
           }

           long[] adder = new long[31];
           for(int i=1; i<=30; i++) {
               adder[i] = adder[i-1];
               if(flag[i]) adder[i] += (1<<(i-1));
           }
           for(int i=0; i<N; i++) {
               int pow = log(a[i]);
               a[i] += adder[pow];
           }
           for(int i=0; i<N; i++) System.out.print(a[i] + " ");
           System.out.println();
        }
    }
    static int log(long a) {
        int count = 0;
        while(a % 2 == 0) {
            ++count;
            a = a / 2;
        }
        return count;
    }
}

The time complexity comes out to be O(Q + NlogM), M = 30 at max.
