Codeforces Round 504 Editorial
By MikeMirzayanov, 8 years ago,

1023A - Single Wildcard Pattern Matching

If there is no wildcard character in the string ss, the answer is "YES" if and only if strings ss and tt are equal. In the other case let's do the following thing: while both strings are not empty and their last characters are equal, let's erase them. Then do the same for the first characters, i.e. while both strings are not empty and their first characters are equal, let's erase them. Now if ss is empty or s=s="*" the answer is "YES", otherwise the answer is "NO".

Problem idea: MikeMirzayanov, awoo; prepared by: vovuh.

1023B - Pair of Toys

The problem is to calculate the number of ways to choose two distinct integers from 11 to nn with sum equals kk.

If k≤nk≤n then the answer is ⌊k2⌋⌊k2⌋ because this is the number of ways to choose two distinct integers from 11 to k−1k−1 with the sum equals kk. Otherwise let mn=n−kmn=n−k will be the minimum possible term in the correct pair of integers. Also let mx=nmx=n will be the maximum possible term in the correct pair of integers. Then the answer is max(0,⌊mx−mn+12⌋)max(0,⌊mx−mn+12⌋) because this is the number of ways to choose two distinct integers from mnmn to mxmx with the sum equals kk.

Problem idea: MikeMirzayanov, prepared by: MikeMirzayanov.

1023C - Bracket Subsequence

Let the array usedused of nn boolean values describe if the corresponding bracket of string ss is included in answer or not.

The algorithm goes like this: iterate over the string from 11 to nn, maintain the stack of positions of currenly unmatched opening brackets stst. When opening bracket is met at position ii, push ii to stst, and when closing bracket is met, set used[st.top()]=Trueused[st.top()]=True, used[i]=Trueused[i]=True and pop the top of stst. When kk values are marked True in usedused then break, iterate from 11 to nn and print the brackets at positions where used[i]=Trueused[i]=True.

Obviously, this algorithm produces a subsequence of ss of length kk. Why will it be a regular bracket sequence? The requirements for it are:

  1. no prefix contains more closing brackets than opening ones;
  2. the total number of closing bracket equals the total number of opening brackets.

The first requirement is met by construction, we couldn't pop more elements from the stack than it had. The second requirement is also met as we marked brackets in pairs.

Overall complexity: O(n)O(n).

Problem idea: MikeMirzayanov, prepared by: awoo.

1023D - Array Restoration

Let's firstly solve the problem as if there are no zeroes in the given array. Let lili be the leftmost occurrence of ii in the array and riri be the rightmost occurrence of ii. The main observation is that you can choose segments (li;ri)(li;ri) for the corresponding queries and this sequence will be correct if and only if there exists any answer for the given array. If there is no occurrence of some value in the array then you can put the segment for it under the segment of some greater value. If there is no occurrence of qq in the array then the answer doesn't exist.

For sure, segment (li,ri)(li,ri) is the minimum segment you can choose. You can expand it in both directions but it will never matter: these positions either will get covered by the segments of greater values or will replace the smaller values with ii (and turn YES to NO if the answer existed). Finally, each position will be covered as each element is a left bound of a segment, a right bound of a segment or just covered by the segment of its value.

That problem can be solved with any data structure that allows you to assign values on segment and get the value of every position (segment tree, sqrt decomposition).

However, all the queries are performed offline (the resulting values are only needed after the queries) and the operation can be replaced with assigning maximum of the current value of element and the value of the query.

This can also be done using set. For each position you should keep the segments which start there and end there. For each segment (li,ri)(li,ri) you push ii to the list for lili and push ii to the list for riri. Now you iterate from 11 to nn, when entering ii you add all values of opening segments to the set, assign the element at position ii the maximum value of the set and remove all the values of closing segments from the set. The complexity of this algorithm is O(nlogq)O(nlog⁡q).

This algorithm can be easily applied to the problem with zeroes in the array. At the beginning you fill the resulting array with ones. After you performed the algorithm with set on the values from 11 to qq, while constructing the segments from the non-zero elements of the given array, you check if the values you assigned are less or equal than the corresponding values of the given array. If that holds then the resulting array is already the correct one. Otherwise the answer doesn't exist. The only corner case there is if no value qq was in array and there were some zeroes. That way you should just change any zero to qq.

Overall complexity: O(nlogq)O(nlog⁡q).

Problem idea: vovuh, MikeMirzayanov; prepared by: awoo.

1023E - Down or Right

Hint: Move from (1,1)(1,1) to the middle by asking queries 'query(row, col, n, n)', starting with row=col=1row=col=1. Similarly, move from (n,n)(n,n) to the middle by asking queries 'query(1, 1, row, col)'. How to ensure that we will meet in the same cell in the middle?

The unusual condition in this problem is (r2−r1)+(c2−c1)≥n−1(r2−r1)+(c2−c1)≥n−1 that must be satisfied in every query. Without it, the following simple code would solve the problem:

int row = 1, col = 1;
while(row < n || col < n) {
  if(row + 1 <= n && query(row + 1, col, n, n)) row++; // move down
  else col++; // move right
}

This program starts in (1,1)(1,1) and greedily moves down if after this move we could still reach the cell (n,n)(n,n). Otherwise, it must move right.

But in this problem, we can only get half the way this method. We must stop at the antidiagonal (one of cells: (1,n),(2,n−1),…,(n,1)(1,n),(2,n−1),…,(n,1)). So maybe it's a good idea to move backward from (n,n)(n,n) the same way, and meet at the antidiagonal? Not really :(

We indeed can apply the same algorithm starting from (n,n)(n,n) and going towards (1,1)(1,1), but possibly we will end in a different cell in the middle. It's possible even for an empty grid, without any blocked cells. We are very close to a correct solution, but let's focus on a thought process for a moment. Analyzing some other possible approach might help with that.

The limitation about the distance at least n−1n−1 is just enough to ask a query between (1,1)(1,1) and a cell from the antidiagonal, and also that cell and (n,n)(n,n).

for(int row = 1; row <= n; ++row) {
  int col = n + 1 - row;
  if(query(1, 1, row, col) && query(row, col, n, n)) {
    print(row, col);
  }
}

This pseudocode would print reasonable candidates for a middle cell in our path — a cell reachable from (1,1)(1,1) and from which the (n,n)(n,n) is reachable. But after choosing some cell like this, it isn't that easy to find a way to the corner cells.

In our first idea, we were able to get from (1,1)(1,1) to one of the reasonable candidates and from (n,n)(n,n) to one of the reasonable candidates, but maybe a different one. Now a very important observation is: the first piece of code in this editorial will reach the leftmost (equivalently: downmost) reasonable candidate, because we always prefer moving down instead of right. For example, in an empty we would reach the bottom left corner. Now we see that we need to guarantee the same when moving from (n,n)(n,n). So we should prioritize left direction over the up direction:

row = n, col = n;
while(row + col > n + 1) {
  if(query(1, 1, row, col - 1)) col -= 1; // move left
  else row -= 1; // move up
}

Problem idea: Errichto, prepared by: Errichto.

1023F - Mobile Phone Network

Consider the forest of your edges.

Let's add edges into this forest greedily from our competitor's set in order of increasing weight until we have a spanning tree. Remember, we don't need to sort, the input is already given in sorted order.

Now, consider the competitor's edges one by one by increasing weight. Since we have a spanning tree, each competitor edge spans some path on our tree.

To process a competitor's edge, we can collapse all the nodes in the path into one single large node, and fix the cost of our edges along this path to be equal to the cost of the currently considered competitor's edge. We know this is an upper bound on cost since if any were higher, then by the cycle property of MSTs, the customer can ignore one of the higher cost edge on this cycle and choose the customer's edge instead.

In addition, this still allows us to satisfy the condition of the customer choosing all of our edges, since by Kruskal's algorithm, the customer will try to greedily add our edges first into their spanning tree and won't be able to add the customer edge since it now forms a cycle). This shows that this cost is the maximum possible profit since we have an upper bound on each edge, and this upper bound also gives a valid solution.

To determine if our answer can be unbounded, we can check if any of our edges remains uncollapsed after processing all competitor edges.

To do this efficiently, we can just do it naively (even without needing to compute LCAs!).

To show it's already fast enough, there are only a total of at most nn merges overall and each merge can be done in α(n)α(n) time, where α(n)α(n) is the inverse ackermann function, so the overall complexity is O(m+(n+k)α(n))O(m+(n+k)α(n)). Unfortunately, this is too hard to separate from solution with logarithms, so those were also allowed to pass (though TL might have been a bit more strict in those cases).

Problem idea: Lewin, prepared by: Lewin.

1023G - Рыбы

First, how do we find the answer in any time complexity? Let us construct a partially ordered set where each element is a single fish in one of the observations. For two elements x=(v1,d1)x=(v1,d1) and y=(v2,d2)y=(v2,d2) we put x<yx<y if xx and yy could possibly be two occurences of the same fish one after the other, that is, there is enough time to get from one observation to another, so that ρ(v1,v2)≤d2−d1ρ(v1,v2)≤d2−d1, where ρ(v1,v2)ρ(v1,v2) is the distance between the tree vertices, and d1d1 and d2d2 are respective day numbers of the two observations. We can now see that the answer is the smallest number of chains needed to cover all the vertices (a chain is a set of pairwise comparable elements of a p.o.set). By Dilworth's theorem, this is equal to the size of the largest antichain (a set of pairwise incomparable elements). Clearly, if a largest antichain contains a fish from an observation, than it must include all fish from this observation as well. We can now solve the problem in something like O(2k)O(2k) time by trying all subsets of observations.

To get a better complexity we need to use the tree structure. To start, how do we check if a set of observations forms an antichain? Here's one criterion that works:

Proposition. A set SS of observations is an antichain iff:

  • For each subtree of the root vv, a subset of SS lying in that subtree is an antichain.
  • There exists a time moment TT such that it is impossible to reach vv at time TT from any observation in SS (possibly travelling back in time).

Indeed, if both of these conditions hold, than for any two observations (v1,d1)(v1,d1) and (v2,d2)(v2,d2) happening in different subtrees we have ρ(v1,v2)=ρ(v,v1)+ρ(v,v2)>|d1−T|+|d2−T|≥|d1−d2|ρ(v1,v2)=ρ(v,v1)+ρ(v,v2)>|d1−T|+|d2−T|≥|d1−d2|, hence these observations are incomparable. On the other hand, if SS is an antichain, for any observation (vi,di)∈S(vi,di)∈S there is an interval of time moments (li,ri)(li,ri) such that we cannot reach the root at any of times in (li,ri)(li,ri). If two observations ii and jj are incomparable, then(li,ri)∩(lj,rj)≠∅(li,ri)∩(lj,rj)≠∅. But since we have a set of intervals with pairwise non-empty intersections, they must all share a common point, hence our time moment TT exists.

Note that in some cases TT is necessarily a non-integer: if there are two vertices at distance 11 from the root, and there are observations happening at days 11 and 22 respectively at these vertices, than any T∈(1,2)T∈(1,2) will suffice. However, TT can always be either an integer or a half-integer, hence we can multiply all times and distances by 22 and assume TT is integer.

The solution is going to be subtree dynamic programming dpv,tdpv,t — the total weight of the largest antichain in the subtree of vv such that no observation can reach vv at time tt. Recalculation is a bit tricky, so let's go step by step. First, if we have values of dpv,tdpv,t, and no observations take place at vv, how do we change dpv,tdpv,t as we travel up the edge to the top of vv? If the length of this edge is ll, then the new values dp′v,tdpv,t′ satisfy dp′v,t=maxt+lt′=t−ldpv,t′dpv,t′=maxt′=t−lt+ldpv,t′. Let's call this procedure "advancing ll steps forward".

How do we account for observations at vv? For an observation (v,t)(v,t) with ff fishes we cannot simply add ff to dpv,tdpv,t since we can obviously reach vv at time tt from this observation. Instead, the correct procedure is:

  • advance the values 11 step;
  • apply dp′v,t=max(dp′v,t,dpv,t+f)dpv,t′=max(dpv,t′,dpv,t+f) for all observations;
  • advance the new values l−1l−1 steps.

Finally, to combine the answers from the subtrees u1,…,uku1,…,uk of a vertex vv (after the advancements were made in them), we simply add them element-wise for each tt: dpv,t=∑uidpui,tdpv,t=∑uidpui,t.

Of course, we cannot store the DP values themselves since there are too much of them to store. Instead, we will need to use a data structure that will maintain a piecewise constant function. More specifically, let us maintain the positions where the function changes values: for each tt such that dpv,t≠dpv,t+1dpv,t≠dpv,t+1 we store the pair (t,dpv,t+1−dpv,t)(t,dpv,t+1−dpv,t). How all of the above manipulations look like in terms of this difference structure?

  • Element-wise sum of two functions is simply the union of the difference sets. (We may want to combine the differences with the same tt value, but for most implementations this is not necessary)
  • When we advance ll steps forward, the borders with positive differences will more ll units to the left, and negative differences will move to the right. When two borders collide, then some of them get eliminated. Consider the function 10,1,510,1,5, with two differences (1,−9)(1,−9) and (2,4)(2,4). After advancing this one step, the new function is 10,10,510,10,5, so there is now a single difference (2,−5)(2,−5). If we keep track of when and where these collisions take place, we can make changes to the structure accordingly.

The structure of choice here is a treap of differences. When we add two functions, we insert all elements of the smaller treap into the larger one; a standard argument shows that at most O(klogk)O(klog⁡k) insertions will take place overall. To keep track of collisions, we'll have to maintain collision times for all adjacent border pairs, and store the minimal value in the root of each subtree. This may seem cumbersome, but every adjacent pair of elements occurs either as (the rightmost vertex in the left subtree of vv, vv), or (vv, the leftmost vertex in the right subtree of vv), hence the standard relax function in the treap implementation can handle this.

One further detail is that we don't want to change the time values in our treaps. Instead, we'll use "lazy" time shifting, that is, we'll keep an additional value ΔΔ associated with each treap, and assume that for entries (t,d)(t,d) with d>0d>0 the actual time should be t−Δt−Δ, and with d<0d<0 the time should be t+Δt+Δ. This way, we won't need to actually change anything in the treap when advancing (apart from resolving collisions).

The total complexity of this solution is O(n+klog2k)O(n+klog2⁡k).

Problem idea: Endagorion, prepared by: Endagorion.

    Well, it is possible to use only STL data structures to solve the problem.

    To do this, one needs to keep the bounds of the segments and the difference of f values between neighboring segments. Also, a trick is needed to account an observation, because it is not a simple addition, it is a max among addition to the previous value and advance operation (and we can't obtain the previous value, since STL set does not support range operations): fnew(t) = max(fold(t) + obs, fold(t - 1), fold(t + 1)). Actually, problem D can be easily solved with O( n + q ). Here you can take a look at the code. The idea is to use 2 pointers, so we can check if the array (except zeros) is correct. How? Initialize array of bools b[i] which tells us if we can find number i further. Then we move right pointer r until a[i - 1] > a[i]. Then we move left pointer l up to r and update bools array if a[l] > a[i] because if we meet number a[l] further, then the segment should have involved index i, so a[i] can't be lower than a[l]. So in this case b[a[l]] becomes false.

In the end we deal with max number (if it's not in the array, then use one zero to get it. No zeros in this case -> the answer is "No") and finally make all other zeroes like previous or following number, it won't break correct array.

The size of array b[i] is q and algorithm is linear in n, so time is really O( n + q ). Moreover it can be upgraded to amortized O( n ) if hash table is used istead of array of bools

        Yes, it sounds like your understanding is correct. We can go over the first sample for explanation.

        For example, the tree initially has edges 1-2, 3-4, 1-3.

        Now, the first considered edge is 2 3 with cost 3, which spans the path 2-1, 1-3. We can fix both of these edges to have cost 3, and also collapse nodes 1,2,3.

    Imagine we have filled some segment with 7s, and there are 9 queries. It is impossible to replace any of the 7s with a number smaller than 7 as all the numbers we can make some element equal to until the end are greater than 7 (in this case, 8 and 9).
    This means that if there is some number i such that between the first and last occurrence of i (which must be the left and right segment of the query that made some elements equal to i, since only that query can make elements equal to i), there is a number j < i, we consider the array to be unobtainable (this is the impossible case).
    The editorial solution checks whether this is possible by checking the smallest element in the aforementioned range (the RMQ technique) and if it is  < i, the array is unobtainable, and if it is not, then there is no earlier query that "interrupts" the ith one.
    If there is no earlier query that interrupts any query (from 1 to q), the array is obtainable via a greedy (or some other) strategy.

        I don't understand what you're asking so I'll cover some cases. Note that it is possible to just change one number in one of the queries.
        {3, 3, 3, 1, 2} is possible. You start from all zeroes, make one number equal to one, one number equal to two, and set the rest to three.

        {3, 3, 3, 1, 2, 3, 1, 2} is not possible as there is a 1 between the first and last occurrence of 3, which is not possible (read my comment above for more clarification).
        Of course this is all under the assumption that there were 3 queries. If there were more, then earlier queries overlapped later ones which is impossible, meaning the answer would be "NO" for all of the cases above (if q > 3).

          You may use fenwick trees. Initialize the fenwick tree array, say fen with 0s. Keep on updating the tree with 1s for each index of query no. i in the given array for 1<=i<=q.

          To check if a smaller number appears between two ends, say l and r, compute diff = fen[r]-fen[l]. If diff > 0, you have a smaller number in between.
