1901A - Line Trip

Idea: BledDest

Tutorial

1901A - Line Trip

We can iterate over the volume of the gas tank from 11 to ∞∞ (in fact 200200 is enough due to problem limitations) and check whether it's enough to travel from the point 00 to the point xx, and back.

Let the volume be VV, then all the following inequalities (which correspond to the ability to travel from the current gas station, having a full tank, to the next one) must be met: a1−0≤Va1−0≤V, a2−a1≤Va2−a1≤V, ..., an−an−1≤Van−an−1≤V, and 2(x−an)≤V2(x−an)≤V (the multiplier 22 because there is no gas station at xx, and we have to go from anan to xx and back without refueling). If all these conditions hold, then VV can be the answer.

However, we can notice that the minimum value of VV that is sufficient for the travel is just the maximum of the left sides of inequalities written above. In other words, the answer to the problem is max(a1,a2−a1,…,an−an−1,2(x−an))max(a1,a2−a1,…,an−an−1,2(x−an)).

Solution (Neon)
#include <bits/stdc++.h>
 
using namespace std;
 
int main() {
  ios::sync_with_stdio(false); cin.tie(0);
  int t;
  cin >> t;
  while (t--) {
    int n, x;
    cin >> n >> x;
    int prev = 0, ans = 0;
    for (int i = 0; i < n; ++i) {
      int a; cin >> a;
      ans = max(ans, a - prev);
      prev = a;
    }
    ans = max(ans, 2 * (x - prev));
    cout << ans << '\n';
  }
}

1901B - Chip and Ribbon

Idea: Roms

Tutorial

1901B - Chip and Ribbon

At first, let's change the statement a bit: instead of teleporting our chip into cell xx, we create a new chip in cell xx (it means that the chip does not disappear from the cell where it was located). And when we want to move a chip, we move any chip to the next cell. Then, cici will be the number of times a chip appeared in the cell ii, and the problem will be the same: ensure the condition on each cici by "creating" the minimum number of chips.

Let's look at value of c1c1. If c1>1c1>1, we have to create at least c1−1c1−1 new chips in cell 11. Let's create that number of chips in that cell.

Then, let's see how we move chips from the cell ii to the cell (i+1)(i+1). If ci≥ci+1ci≥ci+1, then all chips that appeared in the cell (i+1)(i+1) could be moved from the ii-th cell, so we don't need to create any additional chips in that cell.

But if ci<ci+1ci<ci+1, then at least ci+1−cici+1−ci chips should be created in the cell (i+1)(i+1), since we can move at most cici chips from the left.

So, for every ii from 22 to nn, we have to create max(0,ci−ci−1)max(0,ci−ci−1) chips in the ii-th cell; and the number of times we create a new chip in total is c1−1+∑i=2nmax(0,ci−ci−1)c1−1+∑i=2nmax(0,ci−ci−1).

Solution (Roms)
#include <bits/stdc++.h>
 
using namespace std;
 
const int N = 200'000;
 
int t;
 
int main() {
    cin >> t;
    for (int tc = 0; tc < t; ++tc) {
        int n;
        cin >> n;
        vector <int> cnt(n);
        long long res = 0;
        int cur = 0;
        for (int i = 0; i < n; ++i) {
            cin >> cnt[i];
            if (cnt[i] > cur) 
                res += cnt[i] - cur;
            cur = cnt[i];
        }
        
        cout << res - 1 << endl;
    }
    return 0;
}

1901C - Add, Divide and Floor

Idea: BledDest

Tutorial

1901C - Add, Divide and Floor

Sort the array. Notice how applying the operation doesn't change the order of the elements, regardless of xx. It means that it's enough to make the initial minimum and maximum equal to make all elements equal.

Consider the difference between the minimum and the maximum values. What happens to it after an operation? Let the minimum be aa and the maximum be bb. Then it's ⌊b+x2⌋−⌊a+x2⌋⌊b+x2⌋−⌊a+x2⌋. The roundings are difficult to deal with, so let's pretend the parities always align to even. So it's b+x2−a+x2=b−a2b+x2−a+x2=b−a2. Apparently, the difference is just getting divided by 22.

Let's bring the parities back. Notice that the rounding of the difference depends only on the parities of a,ba,b and xx. You can consider all cases of parities of aa and bb to discover that it's always possible to divide the difference by 22, rounding down, and it's never possible to make it less than that.

One easy algorithm to achieve that is the following: if the minimum is even, choose x=0x=0; otherwise, choose x=1x=1. Repeat until the minimum is equal to the maximum.

Overall complexity: O(n+logA)O(n+log⁡A) per testcase.

Solution (awoo)
for _ in range(int(input())):
	n = int(input())
	a = list(map(int, input().split()))
	x, y = min(a), max(a)
	res = []
	while x != y:
		res.append(x % 2)
		x = (x + res[-1]) // 2
		y = (y + res[-1]) // 2
	print(len(res))
	if len(res) <= n:
		print(*res)

1901D - Yet Another Monster Fight

Idea: BledDest

Tutorial

1901D - Yet Another Monster Fight

Let's start with a naive solution. Let ii be the index of the monster we started with. Let's make sure that we can kill all monsters, starting with monster ii.

Suppose we have chosen spell power xx. How to check if all monsters will be definitely killed? Let's iterate on each monster jj and calculate the minimum possible damage it can get:

  • if i=ji=j, then the jj-th monster will receive exactly xx damage. So, xx should be at least ajaj;
  • if i>ji>j, then the minimum amount of damage the jj-th monster may receive happens when the spell first strikes all monsters to the right of monster ii, and then goes to the left of monster ii. It means that n−jn−j monsters will be struck before the jj-th monster, so the jj-th monster will receive x−(n−j)x−(n−j) damage. So, xx should be at least aj+(n−j)aj+(n−j) for every j<ij<i;
  • and if i<ji<j, then the minimum amount of of damage the jj-th monster may receive happens when the spell first strikes all monsters to the left of monster ii, and then goes to the right of monster ii. It means that j−1j−1 monsters will be struck before the jj-th monster, so the jj-th monster will receive x−(j−1)x−(j−1) damage. So, xx should be at least aj+(j−1)aj+(j−1) for every j>ij>i.

So, if we want to start with the ii-th monster, the minimum spell power we need is max(maxj=1i−1aj+(n−j),ai,maxj=i+1naj+(j−1))max(maxj=1i−1aj+(n−j),ai,maxj=i+1naj+(j−1)). This gives us a solution in O(n2)O(n2), but it can be sped up to O(n)O(n) using prefix and suffix maxima.

Solution (vovuh)
#include <bits/stdc++.h>
 
using namespace std;
 
int main() {
#ifdef _DEBUG
    freopen("input.txt", "r", stdin);
//  freopen("output.txt", "w", stdout);
#endif
    
    int n;
    cin >> n;
    vector<int> a(n);
    for (auto &it : a) cin >> it;
    
    vector<int> pref(n), suf(n);
    for (int i = 0; i < n; ++i) {
        pref[i] = a[i] + (n - i - 1);
        suf[i] = a[i] + i;
    }
    for (int i = 1; i < n; ++i) {
        pref[i] = max(pref[i], pref[i - 1]);
    }
    for (int i = n - 2; i >= 0; --i) {
        suf[i] = max(suf[i], suf[i + 1]);
    }
    
    int ans = 2e9;
    for (int i = 0; i < n; ++i) {
        int cur = a[i];
        if (i > 0) cur = max(cur, pref[i - 1]);
        if (i + 1 < n) cur = max(cur, suf[i + 1]);
        ans = min(ans, cur);
    }
    
    cout << ans << endl;
    
    return 0;
}

1901E - Compressed Tree

Idea: BledDest

Tutorial

1901E - Compressed Tree

We can use dynamic programming on tree to solve this problem. After we're done removing vertices, a vertex will be left in the tree after the compression process if and only if its degree is not 22. So, we can try to choose the number of children for each vertex in dynamic programming, and depending on this number of children, the vertex we're considering is either removed from the tree during the compression, or left in the tree.

But if we want to use dynamic programming, we have to root the tree at some vertex, so the degree of each vertex depends not only on the number of its children we left in the tree, but also on whether there exists an edge leading from the ancestor to that vertex. So, for each vertex, we will consider two values of dynamic programming: the best answer for its subtree if there is an edge leading into it from the parent (i. e. we haven't deleted everything outside of that subtree), and the answer if there is no edge leading from the parent (i. e. we deleted everything outside of that subtree).

Let the first of these two values be dpvdpv — the maximum answer for subtree of vv if there is at least one non-deleted vertex not in the subtree of vertex vv (i.e there is an edge from vertex vv up the tree).

Let's look at the dp transitions, depending on the number of children of vertex vv that are not deleted:

  • 00: dpv=max(dpv,av)dpv=max(dpv,av);
  • 11: dpv=max(dpv,maxudpu)dpv=max(dpv,maxudpu) (avav is not taken into account, because vv has 22 incident edges and will be compressed);
  • at least 22: dpv=max(dpv,av+∑udpu)dpv=max(dpv,av+∑udpu).

Then let us consider the case when the resulting tree is in the subtree of some vertex vv. We can update the global answer depending on the number of children of vertex vv that are not deleted:

  • 00: ans=max(ans,av)ans=max(ans,av);
  • 11: ans=max(ans,av+maxudpu)ans=max(ans,av+maxudpu);
  • 22: ans=max(ans,maxu1,u2dpu1+dpu2)ans=max(ans,maxu1,u2dpu1+dpu2) (avav is not taken into account, because vv has 22 incident edges and will be compressed);
  • at least 33: ans=max(ans,av+∑udpu)ans=max(ans,av+∑udpu).

For convenience, we can calculate auxiliary dynamic programming for children of vertex vv: sumisumi is the maximum sum of dpudpu for ii children. From above written transitions, we can see that, sum3sum3 can store maximum sum for at least 33 children.

Don't forget that we can delete all vertices, so the answer is at least 00.

Solution (Neon)
#include <bits/stdc++.h>
 
using namespace std;
 
using li = long long;
 
const li INF = 1e18;
const int N = 555555;
 
int n;
int a[N];
vector<int> g[N];
li dp[N];
li ans;
 
void calc(int v, int p) {
  vector<li> sum(4, -INF);
  sum[0] = 0;
  for (int u : g[v]) if (u != p) {
    calc(u, v);
    for (int i = 3; i >= 0; --i) {
      sum[min(i + 1, 3)] = max(sum[min(i + 1, 3)], sum[i] + dp[u]);
    }
  }
  dp[v] = -INF;
  for (int j = 0; j < 4; ++j) {
    dp[v] = max(dp[v], sum[j] + (j == 1 ? 0 : a[v]));
    ans = max(ans, sum[j] + (j == 2 ? 0 : a[v]));
  }
}
 
int main() {
  ios::sync_with_stdio(false); cin.tie(0);
  int t;
  cin >> t;
  while (t--) {
    cin >> n;
    for (int i = 0; i < n; ++i) cin >> a[i];
    for (int i = 0; i < n; ++i) g[i].clear();
    for (int i = 0; i < n - 1; ++i) {
      int x, y;
      cin >> x >> y;
      --x; --y;
      g[x].push_back(y);
      g[y].push_back(x);
    }
    ans = 0;
    calc(0, -1);
    cout << ans << '\n';
  }
}

1901F - Landscaping

Idea: adedalic

Tutorial

1901F - Landscaping

Let's say that we are searching for the best line that contains the best segment.

Observation 11: the best line touches at least one vertex of the polyline. Otherwise, you can push it down until it touches polyline.

Observation 22: the best line touches two vertices of the polyline. Suppose it touches only one vertex ii then it's not hard to prove that rotating the line around point ii either clockwise or counterclockwise will decrease the area under the line.

Observation 33: if both points are on the left half (i<j<n2i<j<n2) then rotating the line clockwise around point jj will decrease the area. Analogically, if both points are on the right half (n2<i<jn2<i<j) then rotating the line counterclockwise around ii will decrease the area.

Observation 44: since all points of the polyline should be under the line, then the vertices the line touches are the consecutive vertices on the convex hull of the polyline.

In total, we understand that the pair of points we need forms a segment of the convex hull that crosses a vertical line x=n2x=n2.

Knowing that, what can we do? Let's split all vertices in two halves: "left half" with all vertices to the left of n2n2 and "right half" with all vertices to the right. Note that while we process the first half of "queries" the right half (vertices in [n2,n−1][n2,n−1]) remains the same. If we know how to process the first half of queries, then processing the second half is practically the same thing but in the reverse order.

So, how to find that best segment of the convex hull that crosses x=n2x=n2 efficiently while we work with the left half of vertices? Note that if we look at all segments that connect vertex from the left part with vertex from the right part:

  1. each segment will certainly cross x=n2x=n2 in some real point ycyc;
  2. the segment of the convex hull will have maximum possible ycyc (otherwise, it's not a convex hull segment).

There are around n2n2 segment that crosses n2n2 but we don't need all of them. Let's calculate for each vertex ii (i<n2i<n2) (both the old and new values) the segment that crosses n2n2 and has ycyc maximum possible.

Note that this "maximum possible" segment connects ii with some vertex jj that has to be on the convex hull that was built on the right half (otherwise, we will find convex hull point with higher slope and its ycyc will be bigger).

In other words, for each point from the left, we are searching a tangent line to the convex hull on the right. And we can do it efficiently with binary search while checking some cross products. Let's name that tangent line as t(x,y)t(x,y) where (x,y)(x,y) is a point from the left half.

One more observation: yc=y0+y12yc=y0+y12. You can prove it by looking at the area of trapezoid that formed by segment (0,y0)−(n−1,y1)(0,y0)−(n−1,y1). So, let's define a function f(l)f(l) that takes line ll and returns 2yc2yc (or the answer to the task).

We have all we need to calculate the answer: the answer to the ii-th query is equal to the maximum among f(t(j,bj))f(t(j,bj)) for all j≤ij≤i and f(t(j,aj))f(t(j,aj)) for all i<j<n2i<j<n2.

In total, the solution is next:

  1. build convex hull on vertices [n2,n−1][n2,n−1];
  2. calculate t(i,ai)t(i,ai) with binary search for all i<n2i<n2;
  3. calculate f(t(i,ai))f(t(i,ai)) for all i<n2i<n2 and store suffix maximums among these values;
  4. calculate t(i,bi)t(i,bi) with binary search for all i<n2i<n2;
  5. calculate f(t(i,bi))f(t(i,bi)) for all i<n2i<n2 and store prefix maximums among these values;
  6. calculate the ii-th answer as maximum between the ii-th prefix maximum and the (i+1)(i+1)-th suffix maximum.

In order to solve the task for the right half [n2,n−1][n2,n−1] just reverse arrays aa and bb and swap them, and do the same algorithm above. It's true, because replacing first kk values aiai with bibi in array aa is equivalent to replacing last n−kn−k values bibi with aiai in array bb.

Complexity of the solution is O(nlogn)O(nlog⁡n) because of binary searches. Note that we can do everything in integers, except calculating the values of function ff and taking maximum among ff.

Solution (adedalic)
#include<bits/stdc++.h>
 
using namespace std;
 
#define fore(i, l, r) for(int i = int(l); i < int(r); i++)
#define sz(a) int((a).size())
#define all(a) (a).begin(), (a).end()
 
#define x first
#define y second
 
typedef long long li;
typedef long double ld;
typedef pair<int, int> pt;
 
template<class A, class B> ostream& operator <<(ostream& out, const pair<A, B> &p) {
	return out << "(" << p.x << ", " << p.y << ")";
}
template<class A> ostream& operator <<(ostream& out, const vector<A> &v) {
	fore(i, 0, sz(v)) {
		if(i) out << " ";
		out << v[i];
	}
	return out;
}
 
pt operator+ (const pt &a, const pt &b) {
	return {a.x + b.x, a.y + b.y};
}
pt operator- (const pt &a, const pt &b) {
	return {a.x - b.x, a.y - b.y};
}
li operator *(const pt &a, const pt &b) {
	return a.x * 1ll * b.x + a.y * 1ll * b.y;
}
li operator %(const pt &a, const pt &b) {
	return a.x * 1ll * b.y - a.y * 1ll * b.x;
}
 
const int INF = int(1e9);
const li INF64 = li(1e18);
const ld EPS = 1e-9;
 
int n;
vector<pt> old, nw;
 
inline bool read() {
	if(!(cin >> n))
		return false;
	old.resize(n);
	nw.resize(n);
	fore (i, 0, n) {
		old[i].x = i;
		cin >> old[i].y;
	}
	fore (i, 0, n) {
		nw[i].x = i;
		cin >> nw[i].y;
	}
	return true;
}
 
inline ld getTr(const pt &a, const pt &b) {
	pt tmp = b - a;
	pt v = {-tmp.y, tmp.x};
	ld c = v * a;
	ld y0 = (c - v.x * 0) / v.y;
	ld y1 = (c - v.x * li(n - 1)) / v.y;
	return (y0 + y1);
}
 
vector<pt> hull(const vector<pt> &ps, int l, int r) {
	vector<pt> h;
	fore (i, l, r) {
		while (sz(h) > 1 && (h[sz(h) - 1] - h[sz(h) - 2]) % (ps[i] - h[sz(h) - 1]) >= 0)
			h.pop_back();
		h.push_back(ps[i]);
	}
	return h;
}
 
inline void solve() {
	vector<ld> maxTr(n - 1, 0);
	fore (t, 0, 2) {
		auto h = hull(old, n / 2, n);
 
		auto best = [&](const pt &p) {
			int l = -1, r = sz(h) - 1;
			while (r - l > 1) {
				int mid = (l + r) >> 1;
				if ((h[mid] - p) % (h[mid + 1] - h[mid]) >= 0)
					l = mid;
				else
					r = mid;
			}
			return h[r].x;
		};
		
		vector<ld> bs(n / 2 + 1, 0);
		for (int i = n / 2 - 1; i >= 0; i--) {
			int j = best(old[i]);
			bs[i] = getTr(old[i], old[j]);
			bs[i] = max(bs[i], bs[i + 1]);
		}
		
		ld lans = 0;
		fore (i, 0, n / 2) {
			int j = best(nw[i]);
			lans = max(lans, getTr(nw[i], old[j]));
			maxTr[i] = max({maxTr[i], lans, bs[i + 1]});
		}
		
		reverse(all(old));
		reverse(all(nw));
		swap(old, nw);
		fore (i, 0, n)
			old[i].x = nw[i].x = i;
		
		reverse(all(maxTr));
	}
			
	maxTr.push_back(maxTr.back());
	cout << maxTr << endl;
}
 
int main() {
#ifdef _DEBUG
	freopen("input.txt", "r", stdin);
	int tt = clock();
#endif
	ios_base::sync_with_stdio(false);
	cin.tie(0), cout.tie(0);
	cout << fixed << setprecision(12);
	
	if(read()) {
		solve();
		
#ifdef _DEBUG
		cerr << "TIME = " << clock() - tt << endl;
		tt = clock();
#endif
	}
	return 0;
}

    In addressing the question of why choosing xx as the average between the maximum (bb) and minimum (aa) values doesn't always yield the optimal result in problem C, it is essential to understand how the parity effects the calculations.

    We categorize the scenarios based on the parity of aa, bb, and xx. By brute force, we identify eight possible parity combinations, but let's consider two illustrative examples:

    • When a=2ia=2i, b=2j+1b=2j+1 and x=2k+1x=2k+1, the difference is ⌊(b+x)/2⌋−⌊(a+x)/2⌋=j−i+1⌊(b+x)/2⌋−⌊(a+x)/2⌋=j−i+1.

    • When a=2i+1a=2i+1, b=2jb=2j and x=2k+1x=2k+1, the difference is ⌊(b+x)/2⌋−⌊(a+x)/2⌋=j−i−1⌊(b+x)/2⌋−⌊(a+x)/2⌋=j−i−1.

    In the remaining six cases, the difference is ⌊(b+x)/2⌋−⌊(a+x)/2⌋=j−i⌊(b+x)/2⌋−⌊(a+x)/2⌋=j−i.

    The optimal strategy emerges from these calculations. Matching the parity of xx with aa tends to minimize the difference. This is because the difference doesn't depend on kk and different parity can lead to a larger difference when aa is even.

    Now, regarding your specific question, if a=4i+1a=4i+1 and b=4jb=4j, then choosing xx as the average, x=⌊(a+b)/2⌋=2(i+j)=2kx=⌊(a+b)/2⌋=2(i+j)=2k, does not align with the optimal strategy. In this case, xx is even while aa is odd, and as we've established, matching their parities is crucial for optimality.
