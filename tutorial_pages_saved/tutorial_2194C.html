Codeforces Round #1078 Editorial
By TheEvilBird, 6 weeks ago, translation,

2194A - Lawn Mower

Note that the optimal way to remove the boards is to remove a consecutive w−1w−1 boards and then leave one. Thus, every ww-th board will remain. Therefore, the answer is n−⌊nw⌋n−⌊nw⌋.

2194B - Offshores

Let us assume we want to make the ii-th element the maximum. We will calculate Si=∑j=1…nj≠i⌊ajx⌋Si=∑j=1…nj≠i⌊ajx⌋. Notice that the maximum number of transfers we can make to the ii-th account from all the others is exactly this amount (since the condition y≤xy≤x holds, if we make a transfer from some account uu to some account vv, then ⌊aux⌋⌊aux⌋ will decrease by exactly 11, while ⌊avx⌋⌊avx⌋ will increase by at most 11, so intermediate transfers between accounts will not help increase SiSi). Thus, the problem reduces to learning how to quickly compute SiSi for each ii. To do this, we can calculate Sall=∑j=1n⌊ajx⌋Sall=∑j=1n⌊ajx⌋, then Si=Sall−⌊aix⌋Si=Sall−⌊aix⌋

2194C - Secret message

Let's iterate over dd (divisor of nn) and check if we can obtain a string with informativeness dd. For this, for each 1≤i≤d1≤i≤d, we need to check that there is a common letter at positions i,i+d,i+2d,…i,i+d,i+2d,…. To do this, we will iterate over the letter at position ii and check that this letter is also present at positions i+d,i+2d,…i+d,i+2d,…. For position ii, the check can be done in O(nd⋅k2)O(nd⋅k2), and for all ii, it will work in O(nk2)O(nk2). In total, across all dd, the algorithm will run in O(n⋅d(n)⋅k2)O(n⋅d(n)⋅k2) (where d(n)d(n) is the number of divisors of the number nn, which does not exceed 100100 in the problem constraints). With careful implementation, such a solution can pass all tests, but it can be improved.

The idea: instead of storing kk letters at each position, let's store an array of length 26, where for each letter it is recorded whether it is present at this position or not. Now, for each ii, we can check in O(26⋅nd)O(26⋅nd) whether there is a common letter at positions i,i+d,i+2d,⋯i,i+d,i+2d,⋯ (iterate over one of the letters at position ii and check that it fits for ndnd). Thus, the overall asymptotic complexity of the algorithm will be O(26⋅n⋅d(n))O(26⋅n⋅d(n)).

2194E - The Turtle Strikes Back

Let us start with a basic solution. First, consider some path with the maximum total enjoyment from (1,1)(1,1) to (n,m)(n,m). Observe that Raphael must block one of the cells on this path. For each cell that Raphael may block, we can independently compute the maximum enjoyment that Michelangelo can obtain by moving from (1,1)(1,1) to (n,m)(n,m) while avoiding this cell. This approach is naturally implemented using standard grid dynamic programming and leads to a time complexity of O(nm(n+m))O(nm(n+m)).

To further optimize the solution, fix one of the maximum-enjoyment paths SS from (1,1)(1,1) to (n,m)(n,m) on the original grid. Denote the cells of this path by s1,s2,…,sn+m−1s1,s2,…,sn+m−1. Observe that if Raphael does not block any of these cells, Michelangelo can follow the path SS and obtain the maximum possible enjoyment. Therefore, an optimal move for Raphael always consists of blocking some cell sksk.

Let us consider what happens when the cell (i,j)=sk(i,j)=sk is blocked. Michelangelo has two options:

  • follow the same path SS, losing exactly 2ai,j2ai,j enjoyment, which results in the value maxS−2ai,jmaxS−2ai,j;
  • choose a path from (1,1)(1,1) to (n,m)(n,m) that does not pass through the cell (i,j)(i,j).
The first option can be computed directly. The second requires determining the maximum enjoyment of a path that bypasses the blocked cell.

Observe that a path does not pass through the cell (i,j)(i,j) if and only if it contains some cell (a,b)(a,b) such that either a<ia<i and b>jb>j, or a>ia>i and b<jb<j. This allows us to consider all such cells (a,b)(a,b) as possible bypass points.

We introduce two auxiliary dynamic programming tables:

  • dpS[x][y]dpS[x][y] — the maximum enjoyment on a path from (1,1)(1,1) to (x,y)(x,y);
  • dpT[x][y]dpT[x][y] — the maximum enjoyment on a path from (x,y)(x,y) to (n,m)(n,m).
Then the contribution of a path passing through the cell (a,b)(a,b) is equal to dpS[a][b]+dpT[a][b]−a[a][b]dpS[a][b]+dpT[a][b]−a[a][b].

For each cell sksk, we store the maximum of this value over all valid bypass points and denote it by dp[k]dp[k]. The final answer is then given by mink(max(dp[k],maxS−2a[sk]))mink(max(dp[k],maxS−2a[sk])).

It remains to optimize the computation of the values dp[k]dp[k]. For a fixed cell (i,j)(i,j), it is sufficient to consider only the cells (i+1,q<j)(i+1,q<j) and (p<i,j+1)(p<i,j+1). For these cells, we need to quickly find the maximum value of the expression dpS[x][y]+dpT[x][y]−a[x][y]dpS[x][y]+dpT[x][y]−a[x][y].

Note that such maxima are queried over prefixes of rows and columns, so they can be precomputed in O(nm)O(nm). After that, for each cell (i,j)(i,j), the value of the best bypassing path can be obtained in O(1)O(1) time.

Thus, the final solution runs in O(nm)O(nm) time and uses O(nm)O(nm) memory.

2194F2 - Again Trees... (hard version)

Let dp[v][x]dp[v][x] be the number of ways to select a set of edges from the subtree of this vertex, including the edge from vertex vv to its ancestor, such that the XOR of the cut-off vertices in the subtree (vertices which is not connected to vv now) equals xx. To recompute dpdp, we first need to calculate the XOR convolution of the dpdp values of the children. The XOR convolution of arrays aa and bb of length 2k2k is an array ci=∑aj×bj⊕ici=∑aj×bj⊕i. For now, we will compute it in O(len(a)×len(b))O(len(a)×len(b)). After performing the XOR convolution for all children, we must process the option of removing the edge from vv to its ancestor. For this, we iterate over xx, and if xor_subtree[vv] ⊕⊕ x∈bx∈b, then we need to add dp[v][x]dp[v][x] to dp[v][0]dp[v][0]. Thus, this solution works in O(n×230×230)O(n×230×230), since 0≤x<2300≤x<230, and naive convolution works in O(len(a)×len(b))O(len(a)×len(b)). Of course, we are not satisfied with this time complexity, but note that xx in this dpdp is an XOR combination of the set bb.

Let's choose a basis from the set bb and store dpdp in this basis. Now, the number of possible xx in dpdp is 2k2k, and naive XOR convolution works in 4k4k, so the solution runs in O(n×4k)O(n×4k), which is sufficient to pass F1. There are also several other approaches for solving F1, but they cannot be optimized to solve F2.

The first optimization that improves the complexity is to perform XOR convolution using the Hadamard transform (FWHT). Then, the complexity becomes O(n×2k×k)O(n×2k×k), but this is still insufficient to pass F2.

We want to get rid of the multiplication by kk. One XOR convolution via Hadamard looks like this: compute the Hadamard transform of both arrays, then compute the element-wise multiplication, and finally apply the inverse Hadamard transform. Instead of storing dp[v][x]dp[v][x], let's store hadamard(dp[v][x])hadamard(dp[v][x]), i.e., the Hadamard transform. Its formula is: hadamard(a)[i]=∑aj×(−1)popcount(i AND j)hadamard(a)[i]=∑aj×(−1)popcount(i AND j). Then, convolution is performed in O(2k)O(2k) via element-wise multiplication. Now, the difficulty arises with the second recomputation step, where we wanted to compute the value dp[v][x]dp[v][x] at certain points xx and add it to dp[v][0]dp[v][0].

Let's start with the first step. We need to compute ∑dp[v][bi⊕∑dp[v][bi⊕ xor_subtree[v]]]. It is worth noting that if xor_subtree[vv] cannot be expressed in the basis, then such recomputation is unnecessary. First, let's compute this sum assuming xor_subtree[v] = 0. Let pp be an array of size 2k2k, where the positions corresponding to bibi are set to 1, and all others are zeros. Then, the sum we need to compute is the zero element of the XOR convolution of dp[v]dp[v] and pp. We cannot compute the convolution of dp[v]dp[v] and pp directly, but we can compute q=hadamard(q=hadamard(convolution of dp[v]dp[v] and pp), since this is the element-wise product of hadamard(dp[v])hadamard(dp[v]) and hadamard(p)hadamard(p), and hadamard(p)hadamard(p) can be computed once at the beginning of the program. Now, we are interested in the zero element of the inverse transform of qq, which is simply (∑qi)/2k(∑qi)/2k. Now, recall that we have xor_subtree[vv], but to account for it, it is enough to include in the convolution of dp[v]dp[v] and pp an array where only the position xor_subtree[vv] is set to one. The hadamardhadamard of such an array can be computed in O(2k)O(2k) by definition. Thus, we have computed the required sum in O(2k)O(2k).

Now, we need to compute the new hadamard(dp[v])hadamard(dp[v]) after adding the computed sum to dp[v][0]dp[v][0]. But by the definition of the Hadamard transform, we can find the coefficients with which the added sum contributes to the old value of hadamard(dp[v])hadamard(dp[v]).

Thus, in this solution, the Hadamard transform needs to be explicitly computed only for calculating hadamard(p)hadamard(p), while in the rest, we explicitly compute the necessary coefficients by definition and obtain a solution in O(n×2k+t×2k×k)O(n×2k+t×2k×k), where the second term comes from computing hadamard(p)hadamard(p) for each test dataset.

    This is an interesting idea. My intuitive proof of this idea is as follows: Suppose we have a table with the following condition:

    1 0 1 1 0

    0 1 0 1 1

    1 0 1 0 0

    0 1 0 1 0

    0 0 0 0 1

    We want to put cnt12cnt12 1's into one of the tables (let's say the bottom left one, as you've suggested), and all the rest into the opposite one. Let's try to greedily assemble the bottom left table. It should contain cnt12cnt12 ones. Let's try to take them one by one, taking EXACTLY ONE '1' at a time and not taking anything extra.

    Note that if we take the first unit other than the one closest to the lower left edge (let's say we take {i=0, j=3} as the first unit), then it's clear that we'll capture a very large region, which will most likely contain 1's with a shorter Manhattan distance to the lower left corner.

    If we take 1's one by one, we guarantee that we'll ALWAYS take EXACTLY one '1'. (Consider the operation not as a matrix cut, but as a union of the current submatrix with the submatrix of the current element (see photo)). We can always take a submatrix such that the '1' is its upper right corner. This means there won't be any extra 1's inside.

    (I claim that a similar approach to your idea can be used for the upper right corner.)

now lets assume that in the first part we have k2+mk2+m for some positive mm, then the answer will be:

(k2+m)∗(k−k2−m)=(k+2m2)∗(k−2m2)(k2+m)∗(k−k2−m)=(k+2m2)∗(k−2m2) = k2−4m24k2−4m24 = k24−m2k24−m2 << k24k24

similarly if we assume the answer is k2−mk2−m then the answer will be the same which is strictly less than k24k24.

I solved E a bit differently, here was my thought process:

There's exactly 1 cell on every diagonal we visit for every path from (1,1)(1,1) to (n,m)(n,m). That is, for every cc from 1 to n+mn+m, there is exactly one (x,y)(x,y) on the path where x+y=cx+y=c. The proof is just by noting that every step on this path increments (x,y)(x,y) by exactly 1.

Now let dp1[i][j]dp1[i][j] be the max enjoyment from (1,1)(1,1) to (i,j)(i,j), excluding the enjoyment from the cell (i,j)(i,j). Let dp2[i][j]dp2[i][j] be the max enjoyment from (i,j)(i,j) to (n,m)(n,m), including the enjoyment from the cell (i,j)(i,j). The reason why I'm defining the dp's asymmetrically about counting cell (i,j)(i,j) is so that dp1[i][j]+dp2[i][j]dp1[i][j]+dp2[i][j] is the max enjoyment going through (i,j)(i,j). Both these dps are standard, can be calculated in O(mn)O(mn) time.

Now for the key idea: what's the max enjoyment if you don't go through (i,j)(i,j)? Since you visit exactly 1 cell in each diagonal, you can just find the max enjoyment by going through every cell on the "i+ji+j" diagonal except (i,j)(i,j). This gave me the idea to just store dp1[i][j]+dp2[i][j]dp1[i][j]+dp2[i][j] separately, now if you want to find the max enjoyment by negating i,ji,j, modify the value of dp1[i][j]+dp2[i][j]dp1[i][j]+dp2[i][j] in the "i+ji+j" set by decrementing it by 2∗a[i][j]2∗a[i][j], and find the max value of that set. Do this for every element of the grid and you have your answer.

Yes this answer has an extra log factor because of sets but I think you can remove that too by instead of maintaining sets, storing the max and second max value alone for every diagonal.

You can solve E in a really scuffed way:

Spoiler

Start by finding the maximum sum path using dp. Then we try to find a cell to flip in order to reduce Michelangelo’s maximum achievable score.Such a cell must appear in all previously found optimal paths; otherwise, Michelangelo could choose a path that avoids this cell and keep the same score. Also the reduction of the scores of the paths containing that tile has to be greater than max path score found so far — min path score found so far because else Michelangelo could just take the path with maxscore again but with the flipped piece, leading to no score reduction. To do this, we just greedily try to inverse the piece with the biggest score that is ok with all previous conditions and find the best path for Michelangelo now and add it to our previous paths. Once no cell exists that lies on all collected paths and further decreases Michelangelos maximum score, we stop. I have no idea about the time complexity of this, I am kinda sure it's hackable, but it passed sys tests. Implementation: https://codeforces.com/contest/2194/submission/361998218

I had a slightly different solution for problem E (2194E - The Turtle Strikes Back), that is perhaps slightly simpler, though it has the same complexity.

Consider the cells that lie on an antidiagonal of the grid, for example:

...x
..x.
.x..

Since Michaelangelo's path can only go right or down, each path from top-left to bottom-right must pass through exactly one of the cells on this diagonal (and that is true for each parallel line). We can calculate the value of the optimal path going through each of these cells using the dpS and dpT memos as described in the editorial.

Now, the key point is that we only need to consider the best 2 cells on the diagonal. If the cell through which the maximal path runs has a positive value, then Raphael can invert it to reduce the value of the path going through it. It might still be the best possible path, or maybe Michelangelo will have to switch to the path with the second-best value, but he never has to switch to any of the other paths.

So we can process each diagonal one by one, covering all possible cells, for an O(nm)O(nm) solution.

Here is an approach to F1 that does not use XOR basis.

Since XOR is its own inverse, removing a subgraph that has an XOR sum of bjbj from a subtree uu is equivalent to applying an XOR of bjbj on uu. Thus, we just need to maintain the parity of the number of times we have cut a subgraph with XOR sum bjbj.

Our dp state is dp[u][mask]dp[u][mask], where uu is the subtree and maskmask stores the parity of the number of times bjbj has been cut.

Our transitions can be done with an XOR convolution, so our solution runs in O(n∗4k)O(n∗4k).
